# Section 02 — Database Foundation Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Complexity:** ⚠️ **Medium** — straightforward Postgres + SQLAlchemy work, but the schema-isolation contract is load-bearing and the Alembic safety belt MUST be tested as part of the plan, not assumed.

**Goal:** Establish the `syntrix`-schema database foundation that every later section will build on: a dedicated `syntrix_admin` schema owner and `syntrix_app` runtime role inside the user's shared Supabase Postgres, the four required extensions verified present, SQLAlchemy 2.0 async engine pinned to `search_path = syntrix, public`, and Alembic configured with an `include_object` filter that **raises** on any operation targeting a non-`syntrix` schema.

**Architecture:**

- **Two roles, two URLs.** `syntrix_admin` owns the `syntrix` schema and runs migrations (`DATABASE_ADMIN_URL`). `syntrix_app` is the runtime role with only `USAGE` on the schema and table-level CRUD privileges granted via `ALTER DEFAULT PRIVILEGES` (`DATABASE_URL`). The app never logs in as the admin role; migrations never run as the app role.
- **Bootstrap SQL, not migration.** Role creation, schema creation, and `CREATE EXTENSION` need superuser/admin privileges that `syntrix_admin` does not have. These live in `backend/db/bootstrap.sql`, run once by the user against the Supabase Postgres with `psql` as a superuser. The bootstrap is idempotent (`IF NOT EXISTS`) and can be re-run safely.
- **Alembic safety belt.** `env.py` defines an `include_object` function that **raises `RuntimeError`** (not silently filters) the moment it sees an object outside the `syntrix` schema. A misconfigured autogenerate becomes a loud test failure, never a silent migration that touches `auth.users`. Backed by a unit test that constructs a metadata object spanning `syntrix` and `auth` schemas and asserts the filter raises.
- **`search_path` set per connection.** A SQLAlchemy `connect` event listener executes `SET search_path TO syntrix, public` on every new DBAPI connection. Belt-and-braces with the Postgres role-level `ALTER ROLE syntrix_app SET search_path = syntrix, public` from the bootstrap. Either alone would work; both is defensive.
- **Async runtime, sync migrations.** App code uses `create_async_engine` + `AsyncSession`. Alembic env.py uses a sync `Engine` — migrations are inherently sequential and don't benefit from async. Both built on `psycopg` (psycopg3) so the same DSN format works either way.
- **No tables yet.** This section ships only the foundation. The baseline migration is an empty `op.execute("-- baseline")` that proves Alembic can run end-to-end. `users`, `oauth_identities`, `refresh_tokens`, etc. all land in their owning sections (04 and beyond).
- **Test fixtures.** A `db_session` pytest fixture wraps each test in a connection-level transaction that rolls back on teardown — no test commits leak. A `db_engine` session-scoped fixture builds one async engine per test session.

**Tech Stack:** PostgreSQL 15+ (whatever Supabase ships), SQLAlchemy 2.0 (async), Alembic ≥1.14, psycopg ≥3.2, pydantic-settings ≥2.6.

**Strict rules from `CLAUDE.md` that apply to this section:**

- Every CREATE/ALTER/DROP in every migration targets `syntrix.*` only. If autogen ever produces an op against another schema, that's the safety belt firing — fix the filter or the autogen call, do not edit the migration to "work around" it.
- Inspect every generated migration before applying.
- This branch (`feat/db`) merges to `main` and must be confirmed merged **before** Section 03 starts.

- [ ] **Step 2: Create `backend/db/bootstrap.sql`**

```sql
-- Syntrix database bootstrap.
-- Run ONCE as a Postgres superuser against the Supabase Postgres.
-- Idempotent: re-running is safe.
--
-- Usage:
--   psql "postgresql://postgres:<SUPERUSER_PASSWORD>@127.0.0.1:5432/postgres" \
--        -v syntrix_admin_password="'<SET_ME>'" \
--        -v syntrix_app_password="'<SET_ME>'" \
--        -f backend/db/bootstrap.sql

\set ON_ERROR_STOP on

BEGIN;

-- ---- Extensions (live in the schema where they install themselves;
-- ---- Supabase puts most of these in `extensions` or `public`. They are
-- ---- referenced via search_path; we do not own them.) ----
CREATE EXTENSION IF NOT EXISTS citext;
CREATE EXTENSION IF NOT EXISTS pg_trgm;
CREATE EXTENSION IF NOT EXISTS ltree;
CREATE EXTENSION IF NOT EXISTS pgcrypto;

-- ---- Roles ----
DO $$
BEGIN
  IF NOT EXISTS (SELECT 1 FROM pg_roles WHERE rolname = 'syntrix_admin') THEN
    CREATE ROLE syntrix_admin LOGIN PASSWORD :syntrix_admin_password;
  ELSE
    EXECUTE format('ALTER ROLE syntrix_admin WITH LOGIN PASSWORD %L',
                   :'syntrix_admin_password');
  END IF;

  IF NOT EXISTS (SELECT 1 FROM pg_roles WHERE rolname = 'syntrix_app') THEN
    CREATE ROLE syntrix_app LOGIN PASSWORD :syntrix_app_password;
  ELSE
    EXECUTE format('ALTER ROLE syntrix_app WITH LOGIN PASSWORD %L',
                   :'syntrix_app_password');
  END IF;
END
$$;

-- ---- Schema ----
CREATE SCHEMA IF NOT EXISTS syntrix AUTHORIZATION syntrix_admin;

-- ---- Schema-level grants ----
-- syntrix_admin owns the schema (set above) and gets full rights implicitly.
-- syntrix_app needs USAGE to reference objects inside; CREATE is deliberately
-- withheld so the runtime role cannot define tables/indexes.
GRANT USAGE ON SCHEMA syntrix TO syntrix_app;

-- ---- Default privileges ----
-- When syntrix_admin creates a new table/sequence in `syntrix`, syntrix_app
-- automatically receives CRUD on the table and USAGE+UPDATE on the sequence.
-- This is what makes "syntrix_admin runs migrations; syntrix_app runs the app"
-- ergonomic — we never have to re-grant after each migration.
ALTER DEFAULT PRIVILEGES FOR ROLE syntrix_admin IN SCHEMA syntrix
  GRANT SELECT, INSERT, UPDATE, DELETE ON TABLES TO syntrix_app;
ALTER DEFAULT PRIVILEGES FOR ROLE syntrix_admin IN SCHEMA syntrix
  GRANT USAGE, SELECT, UPDATE ON SEQUENCES TO syntrix_app;
ALTER DEFAULT PRIVILEGES FOR ROLE syntrix_admin IN SCHEMA syntrix
  GRANT EXECUTE ON FUNCTIONS TO syntrix_app;

-- ---- Per-role search_path ----
-- Belt to the per-connection braces SQLAlchemy will also apply.
ALTER ROLE syntrix_admin IN DATABASE postgres SET search_path = syntrix, public;
ALTER ROLE syntrix_app   IN DATABASE postgres SET search_path = syntrix, public;

-- ---- Defensive: revoke CREATE on public from PUBLIC so syntrix_app can't
-- ---- accidentally write to the shared public schema. ----
-- (No-op on most Supabase setups where this is already revoked by template;
-- safe to run regardless.)
REVOKE CREATE ON SCHEMA public FROM PUBLIC;

COMMIT;
```

```python
from functools import lru_cache

from pydantic import Field
from pydantic_settings import BaseSettings, SettingsConfigDict


class Settings(BaseSettings):
    model_config = SettingsConfigDict(
        env_file=".env",
        env_file_encoding="utf-8",
        extra="ignore",
        case_sensitive=False,
    )

    pg_host: str = Field(default="127.0.0.1")
    pg_port: int = Field(default=5432)
    pg_db: str = Field(default="postgres")
    syntrix_app_password: str
    syntrix_admin_password: str

    @property
    def database_url(self) -> str:
        return (
            f"postgresql+psycopg://syntrix_app:{self.syntrix_app_password}"
            f"@{self.pg_host}:{self.pg_port}/{self.pg_db}"
        )

    @property
    def database_admin_url(self) -> str:
        return (
            f"postgresql+psycopg://syntrix_admin:{self.syntrix_admin_password}"
            f"@{self.pg_host}:{self.pg_port}/{self.pg_db}"
        )


@lru_cache
def get_settings() -> Settings:
    return Settings()
```

The Settings class lives in `backend/` working dir at runtime (the FastAPI process runs from `backend/`); the `.env` file is one level up at the repo root. For host-run tests we'll point pydantic-settings at the repo-root `.env` via env var; see Task 7.

```python
from sqlalchemy import MetaData
from sqlalchemy.orm import DeclarativeBase


# Naming convention so Alembic autogenerate produces stable, predictable
# constraint names (essential for diff-style migrations).
NAMING_CONVENTION = {
    "ix": "ix_%(column_0_label)s",
    "uq": "uq_%(table_name)s_%(column_0_name)s",
    "ck": "ck_%(table_name)s_%(constraint_name)s",
    "fk": "fk_%(table_name)s_%(column_0_name)s_%(referred_table_name)s",
    "pk": "pk_%(table_name)s",
}

metadata_obj = MetaData(schema="syntrix", naming_convention=NAMING_CONVENTION)


class Base(DeclarativeBase):
    """Declarative base. Every model inherits from this so its table lives
    inside the `syntrix` schema and uses the shared naming convention."""

    metadata = metadata_obj
```

The schema is set on the metadata, not per-table. Every `Base` subclass inherits this automatically.

- [ ] **Step 3: Create `backend/app/db/engine.py`**

```python
from sqlalchemy import event
from sqlalchemy.ext.asyncio import AsyncEngine, create_async_engine

from app.config import get_settings


def _build_async_engine() -> AsyncEngine:
    settings = get_settings()
    engine = create_async_engine(
        settings.database_url,
        echo=False,
        pool_pre_ping=True,
        pool_size=10,
        max_overflow=5,
    )

    @event.listens_for(engine.sync_engine, "connect")
    def _set_search_path(dbapi_conn, _conn_record):
        # Belt-and-braces with `ALTER ROLE … SET search_path` in bootstrap.sql.
        # Either alone would work; both is defensive.
        cursor = dbapi_conn.cursor()
        try:
            cursor.execute("SET search_path TO syntrix, public")
        finally:
            cursor.close()

    return engine


async_engine: AsyncEngine = _build_async_engine()
```

- [ ] **Step 4: Create `backend/app/db/session.py`**

```python
from collections.abc import AsyncIterator

from sqlalchemy.ext.asyncio import AsyncSession, async_sessionmaker

from app.db.engine import async_engine


async_session_factory = async_sessionmaker(
    bind=async_engine,
    expire_on_commit=False,
    autoflush=False,
    autocommit=False,
)


async def get_session() -> AsyncIterator[AsyncSession]:
    """FastAPI dependency: yields a session, commits on success, rolls back on
    exception, always closes."""
    async with async_session_factory() as session:
        try:
            yield session
            await session.commit()
        except Exception:
            await session.rollback()
            raise
```

```python
if config.config_file_name is not None:
    fileConfig(config.config_file_name)

# Inject the admin DSN at runtime so secrets stay out of alembic.ini.
config.set_main_option("sqlalchemy.url", get_settings().database_admin_url)

target_metadata = Base.metadata


SAFE_SCHEMA = "syntrix"


def include_name(name: str | None, type_: str, parent_names: dict[str, str | None]) -> bool:
    """Restrict reflection to the syntrix schema during autogenerate."""
    if type_ == "schema":
        return name == SAFE_SCHEMA
    return True


def include_object(object, name, type_, reflected, compare_to):
    """SAFETY BELT.

    Refuses (raises) on any object whose schema is not 'syntrix'. A normal
    filter would silently skip; a raise turns a misconfigured autogenerate
    into a loud test failure instead of a silent migration that touches
    auth.users.
    """
    obj_schema = getattr(object, "schema", None)
    if obj_schema is not None and obj_schema != SAFE_SCHEMA:
        raise RuntimeError(
            f"Refusing to operate on schema {obj_schema!r} "
            f"(object {name!r}, type {type_!r}). "
            f"Only schema {SAFE_SCHEMA!r} is permitted."
        )
    return True


def run_migrations_offline() -> None:
    url = config.get_main_option("sqlalchemy.url")
    context.configure(
        url=url,
        target_metadata=target_metadata,
        literal_binds=True,
        dialect_opts={"paramstyle": "named"},
        version_table="alembic_version",
        version_table_schema=SAFE_SCHEMA,
        include_schemas=True,
        include_name=include_name,
        include_object=include_object,
        compare_type=True,
        compare_server_default=True,
    )
    with context.begin_transaction():
        context.run_migrations()


def run_migrations_online() -> None:
    connectable = engine_from_config(
        config.get_section(config.config_ini_section, {}),
        prefix="sqlalchemy.",
        poolclass=pool.NullPool,
    )

    with connectable.connect() as connection:
        # search_path for migrations — same belt as runtime engine.
        connection.exec_driver_sql("SET search_path TO syntrix, public")
        context.configure(
            connection=connection,
            target_metadata=target_metadata,
            version_table="alembic_version",
            version_table_schema=SAFE_SCHEMA,
            include_schemas=True,
            include_name=include_name,
            include_object=include_object,
            compare_type=True,
            compare_server_default=True,
        )
        with context.begin_transaction():
            context.run_migrations()


if context.is_offline_mode():
    run_migrations_offline()
else:
    run_migrations_online()
```

```python
import pytest
from sqlalchemy import Column, Integer, MetaData, Table

# Import the filter directly from env.py. Alembic's env.py is normally only
# run via `alembic` CLI, but the filter is just a function — we can import it.
import importlib.util
import pathlib

_env_path = pathlib.Path(__file__).resolve().parents[3] / "alembic" / "env.py"
_spec = importlib.util.spec_from_file_location("alembic_env", _env_path)
_module = importlib.util.module_from_spec(_spec)
# Don't actually execute env.py — it would try to connect. Instead, read the
# source and exec just the filter definitions in an isolated namespace.
_source = _env_path.read_text()


def _load_filters():
    ns: dict = {"context": None}
    # Strip the trailing `if context.is_offline_mode(): ...` so import doesn't
    # try to run migrations. We only need the function defs.
    safe = _source.split("def run_migrations_offline")[0]
    exec(safe, ns)
    return ns["include_object"], ns["include_name"]


include_object, include_name = _load_filters()


def _table_in_schema(schema: str, name: str = "users") -> Table:
    md = MetaData(schema=schema)
    return Table(name, md, Column("id", Integer, primary_key=True))


def test_include_object_allows_syntrix():
    tbl = _table_in_schema("syntrix")
    assert include_object(tbl, "users", "table", False, None) is True


def test_include_object_raises_on_auth_schema():
    tbl = _table_in_schema("auth")
    with pytest.raises(RuntimeError, match="auth"):
        include_object(tbl, "users", "table", False, None)


def test_include_object_raises_on_public_schema():
    tbl = _table_in_schema("public")
    with pytest.raises(RuntimeError, match="public"):
        include_object(tbl, "anything", "table", False, None)


def test_include_object_allows_unschemed():
    # Objects without a schema attribute (e.g. Index without an explicit
    # schema) should pass; SQLAlchemy will resolve them via metadata.schema.
    class Bare:
        pass

    obj = Bare()
    assert include_object(obj, "ix_anything", "index", False, None) is True


def test_include_name_filters_schemas_to_syntrix_only():
    assert include_name("syntrix", "schema", {}) is True
    assert include_name("auth", "schema", {}) is False
    assert include_name("public", "schema", {}) is False
    # Non-schema names always pass — per-object filtering happens in
    # include_object.
    assert include_name("users", "table", {"schema_name": "syntrix"}) is True
```

## Self-review notes (writer to executor)

- **Safety belt is *tested*, not assumed.** Task 8 deliberately sabotages the filter and confirms the tests catch the regression. Don't skip that step — without it, the test only proves "the function exists," not "it does what it claims."
- **Two roles, two URLs, on purpose.** The runtime role's lack of CREATE on `syntrix` is what forces all DDL through Alembic. If you ever find yourself thinking "let me just `CREATE TABLE` from the app for X," that's a category error — write a migration.
- **No models yet.** Section 02 builds the foundation. `users`, `oauth_identities`, `refresh_tokens` all land in Section 04 (`feat/auth`). `communities` in 06. Etc. If the engineer is tempted to "just add `users` here because auth comes next," push back — that's scope creep across the branch-per-section boundary.
- **`asyncpg` is not used.** psycopg3 supports async natively, the DSN format is unified, and we don't need two drivers. SQLAlchemy 2.0 + psycopg3 is the canonical modern stack.
- **`.env` is the secrets boundary.** Passwords live there only; the bootstrap SQL reads them from `psql -v` variables (which expand from shell env). `pytest` and `make` both inherit shell env, so `export SYNTRIX_APP_PASSWORD=…` once per shell session is enough. CI later will set these from a secrets store.
- **Extensions live outside `syntrix`.** Supabase installs them in `public` or `extensions`. We reference them via `search_path`, but we never `ALTER EXTENSION ... SET SCHEMA syntrix` — that would conflict with other Supabase tenants. The safety belt doesn't fire on extension types because Alembic doesn't autogenerate against them.
- **Transactional test isolation, not schema-per-test.** Every test runs inside a connection-level transaction that rolls back on teardown. Simple, fast, and means a developer can run `pytest` against a populated database without polluting it. The one limitation: tests that need their own explicit commits (very rare) need a different fixture.
- **Compose override pattern.** `.env` holds host-relative values (so `make test`, `make db-migrate` work on the host). `docker-compose.yml` overrides `PG_HOST=db` for the backend container only. This is the cleanest way to handle the "same secret, different hostname" split without templating.
