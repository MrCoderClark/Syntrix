# Section 03 — Design System Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Complexity:** ⚠️ **Medium** — many files, but each is straightforward. The risk is visual drift from the locked mockup, not algorithmic difficulty.

**Goal:** Stand up Syntrix's visual identity so every later section can build UI without reinventing the visual language: fonts via `next/font`, CSS custom property tokens, the paper-grain background, the wordmark, the app shell (sidebar + topbar), a library of UI primitives (Button, IconButton, Avatar, Tab, Input, PageHeader, Card with SVG art, VoteWidget), plus 404 and loading states.

**Architecture:**

- **`next/font/google`** loads Fraunces (display, variable serif with `SOFT` + `opsz` axes), Hanken Grotesk (body, 400–700), and JetBrains Mono (meta/code, 400–500). Font CSS variables are set on `<html>` via className injection.
- **`globals.css`** holds CSS custom properties (colors, typography, spacing, shadows, motion), a minimal reset, the warm-paper background, the SVG noise overlay (`::before` on body, `mix-blend-mode: multiply`), and selection color.
- **CSS Modules** (`*.module.css`) for every component — scoped styles, no class collisions, no Tailwind.
- **Component files** in `frontend/components/` organized by role: `shell/` for layout (Sidebar, Topbar, Shell wrapper), `ui/` for primitives, plus `Wordmark.tsx` at the top level.
- **Reference:** `docs/superpowers/mockups/phase1-home.html` is the authoritative visual spec. Every token value and component style is extracted from that file. When in doubt, open the mockup in a browser and match it.

**Tech Stack:** Next.js 15 (App Router), React 19, TypeScript 5.7, CSS Modules, `next/font/google`.

**Strict rules from `CLAUDE.md` that apply to this section:**

- Frontend must not look AI-generated. No default shadcn theme, no generic SaaS hero, no Inter/Geist. Visual identity is reviewed per section.
- This branch (`feat/design-system`) merges to `main` and must be confirmed merged **before** Section 04 starts.
- No Tailwind utility-flood. Hand-rolled CSS with CSS variables, component-scoped.

import type { ButtonHTMLAttributes, ReactNode } from "react";
import styles from "./Button.module.css";

interface ButtonProps extends ButtonHTMLAttributes<HTMLButtonElement> {
  variant?: "default" | "primary" | "ghost";
  children: ReactNode;
}

export function Button({
  variant = "default",
  className,
  children,
  ...props
}: ButtonProps) {
  const cls = [
    styles.btn,
    variant !== "default" ? styles[variant] : "",
    className,
  ]
    .filter(Boolean)
    .join(" ");

  return (
    <button className={cls} {...props}>
      {children}
    </button>
  );
}

## Self-review notes (writer to executor)

- **Visual reference is authoritative.** The `phase1-home.html` mockup is the single source of truth for every token value, spacing, shadow, border radius, and font setting. The CSS values in this plan were extracted from that file. If something looks off during implementation, open the mockup in a browser and match pixel-for-pixel.
- **CSS Modules, not global classes.** Every component has a `.module.css` file. The only global CSS is in `globals.css` (tokens, reset, keyframes). No component ever references a class from another component's module.
- **No icon library.** Icons are inline SVGs in `icons.tsx`, matching the mockup's hand-drawn SVGs exactly. Don't introduce lucide-react, heroicons, or similar — those carry the AI-generated aesthetic Syntrix actively avoids.
- **Shell is always-on in Phase 1.** The `Shell` wrapper is in `layout.tsx`, so every page gets the sidebar + topbar. If a future page needs a different layout (e.g., `/signin` should probably not have the sidebar), that's a layout-group problem solved in Section 04.
- **`"use client"` only where needed.** `VoteWidget.tsx` is `"use client"` because it has interactive state. Everything else is a server component by default.
- **Card art SVGs use unique `id` attributes.** SVG `<defs>` with `id` conflict when multiple SVGs are on the same page. The `CardArt` component takes an `id` prop and namespaces all internal IDs. This is load-bearing — without it, every card renders the first card's gradients.
- **Animation stagger via `animationDelay` prop**, not nth-child. This is more reliable when the grid is dynamic (filtered, paginated, etc.).
- **Fraunces `SOFT` axis.** The `SOFT` axis controls how round the serifs are (0 = sharp, 100 = soft/rounded). The wordmark uses `SOFT: 80`, the page header glyph uses `SOFT: 100`, the title uses `SOFT: 0` (sharp), the italic accent uses `SOFT: 100` (soft). These values are from the mockup and are load-bearing for the visual identity.
- **No TipTap, no Shiki, no post creation.** The home page showcases static mock data. Real posts land in Section 08. The Card component's props are designed to match the post model from the PRD but don't import any backend types yet.
- **Polish backlog item 4** ("Replace the placeholder 'S' page-header glyph") — the glyph is currently just the letter "S" in a styled square. A real Syntrix mark would be designed separately. Leaving the "S" is fine for Phase 1; the PRD acknowledges it as a polish item.
