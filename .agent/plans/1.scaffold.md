# Section 01 — Scaffold Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Complexity:** ✅ **Simple** — file creation and config wiring. No algorithms, no business logic. Single-pass executable.

**Goal:** Stand up the Syntrix monorepo skeleton so every later section has a place to land code: a Python/FastAPI backend with `/health`, a Next.js frontend rendering a placeholder, both running side-by-side under Docker Compose on the user's existing Supabase network, all started with `make dev`, all gated by pre-commit hooks.

**Architecture:**

- **`backend/`** — Python 3.12 + FastAPI, dependencies managed with `uv` (lockfile-driven), single `app/main.py` exposing `/health`. Pytest for tests, ruff + black for lint/format.
- **`frontend/`** — Next.js 15 (App Router) + React 19 + TypeScript, no Tailwind, minimal placeholder page and a `/health` route that proxies the backend (proves compose wiring without UI). ESLint + Prettier. Real fonts + design tokens land in Section 03.
- **`docker-compose.yml`** — defines two services (`backend`, `frontend`) joined to an *external* Docker network created by the user's already-running Supabase stack. We never modify, redefine, or restart Supabase from this repo.
- **`Makefile`** — the single entry point. `make dev` chains the Supabase compose file with our own via `-f` flags so one command brings up the full loop. Targets: `dev`, `dev-detached`, `down`, `logs`, `backend`, `frontend`, `test`, `lint`, `fmt`, `smoke`, `precommit-install`.
- **`.pre-commit-config.yaml`** — runs ruff, black, prettier, eslint, and the basics (trailing whitespace, EOF newline, large-file guard, merge-conflict marker) on every commit.

**Tech Stack:** Python 3.12, FastAPI ≥0.115, uvicorn, uv, pytest, ruff, black; Next.js 15.1, React 19, TypeScript 5.7, ESLint 9, Prettier 3; Docker, Docker Compose v2, GNU Make, pre-commit.

**Strict rules from `CLAUDE.md` that apply to this section:**

- The Python backend uses a `venv`. (`uv` creates `backend/.venv/` automatically; that satisfies the rule.)
- No `--no-verify` on commits. Pre-commit must pass; if it fails, fix the root cause.
- This branch (`feat/scaffold`) must merge to `main` and be confirmed merged **before** Section 02 starts. Do not open a PR — the user owns commit, push, and merge from here on.

## Task 6 — Docker Compose for Syntrix services

This compose file describes **only** the two Syntrix services. We attach them to the *external* Docker network that the user's Supabase stack already manages, so `backend` can reach Supabase's `db` and `storage` containers without us touching the Supabase compose at all. The Makefile in Task 7 chains both compose files behind a single `make dev`.

**Files:**
- Create: `.env.example`, `docker-compose.yml`
- Local-only (not committed): `.env`

- [ ] **Step 1: Create `.env.example`**

```dotenv
# REQUIRED — absolute path to the user's existing Supabase docker-compose.yml.
SUPABASE_COMPOSE_FILE=/absolute/path/to/your/supabase/docker/docker-compose.yml

# REQUIRED — name of the Docker network the Supabase stack creates.
# Find with: docker network ls | grep -i supabase
SUPABASE_NETWORK=supabase_default

# Placeholder DSN — real connection wiring lands in Section 02 (feat/db).
DATABASE_URL=postgresql+psycopg://syntrix_app:dev@db:5432/postgres

# How the frontend container reaches the backend container inside the compose network.
BACKEND_URL=http://backend:8000
```

- [ ] **Step 2: Create `docker-compose.yml`**

```yaml
name: syntrix

services:
  backend:
    build:
      context: ./backend
    image: syntrix-backend:dev
    container_name: syntrix-backend
    environment:
      - DATABASE_URL=${DATABASE_URL}
    volumes:
      - ./backend/app:/app/app
    ports:
      - "8000:8000"
    networks:
      - default
      - supabase

  frontend:
    build:
      context: ./frontend
    image: syntrix-frontend:dev
    container_name: syntrix-frontend
    environment:
      - BACKEND_URL=${BACKEND_URL}
    volumes:
      - ./frontend:/app
      - /app/node_modules
      - /app/.next
    ports:
      - "3000:3000"
    depends_on:
      - backend
    networks:
      - default

networks:
  supabase:
    name: ${SUPABASE_NETWORK}
    external: true
```

The two anonymous volumes on `frontend` (`/app/node_modules`, `/app/.next`) shadow the bind-mounted host directory so the container's installed `node_modules` and Next build cache aren't clobbered by the host mount.

## Task 7 — Makefile

**Files:**
- Create: `Makefile`

- [ ] **Step 1: Create `Makefile`**

> Indentation **must be tabs**, not spaces — make is strict about this.

```makefile
SHELL := /bin/bash

# Load .env into make's environment if present
ifneq (,$(wildcard .env))
include .env
export
endif

COMPOSE := docker compose -f docker-compose.yml
ifneq (,$(SUPABASE_COMPOSE_FILE))
COMPOSE := docker compose -f $(SUPABASE_COMPOSE_FILE) -f docker-compose.yml
endif

.DEFAULT_GOAL := help

.PHONY: help
help: ## Show this help
	@grep -E '^[a-zA-Z_-]+:.*?## .*$$' $(MAKEFILE_LIST) | awk 'BEGIN{FS=":.*?## "}{printf "  \033[36m%-20s\033[0m %s\n", $$1, $$2}'

.PHONY: dev
dev: ## Bring up Supabase (if path set) + Syntrix backend + frontend
	$(COMPOSE) up --build

.PHONY: dev-detached
dev-detached: ## Same as dev, detached
	$(COMPOSE) up --build -d

.PHONY: down
down: ## Stop Syntrix services (leave Supabase running)
	docker compose -f docker-compose.yml down

.PHONY: logs
logs: ## Tail Syntrix logs
	docker compose -f docker-compose.yml logs -f

.PHONY: backend
backend: ## Run backend tests
	cd backend && uv run pytest -v

.PHONY: frontend
frontend: ## Run frontend lint
	cd frontend && npm run lint

.PHONY: test
test: backend ## Run all tests (currently backend only)

.PHONY: lint
lint: ## Lint backend + frontend
	cd backend && uv run ruff check .
	cd frontend && npm run lint
	cd frontend && npm run fmt:check

.PHONY: fmt
fmt: ## Format backend + frontend
	cd backend && uv run ruff check --fix . && uv run black .
	cd frontend && npm run fmt

.PHONY: smoke
smoke: ## Bring up the stack, curl both health endpoints, tear down
	$(COMPOSE) up --build -d
	@echo "Waiting for services..."
	@for i in $$(seq 1 30); do \
		if curl -fsS http://127.0.0.1:8000/health > /dev/null 2>&1 \
		&& curl -fsS http://127.0.0.1:3000/health > /dev/null 2>&1; then \
			echo "Both up."; break; \
		fi; sleep 2; \
	done
	@echo "--- backend /health ---"
	curl -fsS http://127.0.0.1:8000/health; echo
	@echo "--- frontend /health (proxied) ---"
	curl -fsS http://127.0.0.1:3000/health; echo
	$(MAKE) down

.PHONY: precommit-install
precommit-install: ## Install git hooks
	pre-commit install
```

## Self-review notes (writer to executor)

- **TDD where meaningful.** The backend `/health` endpoint is written test-first (Task 2). The frontend `/health` route and the compose wiring are validated by the `make smoke` integration test in Task 9, not unit tests — composition is what we're verifying, and a unit test of a one-line `fetch` proxy would be theatre.
- **No DB.** This plan does not touch Alembic, the `syntrix` schema, or any SQL. Section 02 (`feat/db`) owns all of that. The `DATABASE_URL` env var in `.env.example` is a placeholder so the env shape is established; nothing reads it yet.
- **No fonts, tokens, or design components.** Section 03 owns the design system. The placeholder home page is deliberately ugly — making it look "designed" before the design tokens land would create a v1 we'd just throw away.
- **No auth.** Section 04 owns OAuth.
- **`uv` is the package manager.** If you prefer plain pip later, swap `[dependency-groups]` for `[project.optional-dependencies]` and replace `uv sync` calls with `pip install -e .[dev]`. The Dockerfile would also need updating. Don't do this swap in Section 01 — `uv` is faster and the lockfile is real.
- **Pinned versions** for Next.js (15.1), React (19.0), Python (3.12), and the `uv` Docker image (0.5.4). If you hit a breaking change later, bump conservatively and note the bump in the commit message.
- **Supabase compose is referenced, not modified.** We never write to `$SUPABASE_COMPOSE_FILE`. The user's other tenants on that stack stay untouched. This is the same principle as the `syntrix`-schema isolation rule, applied to compose.
