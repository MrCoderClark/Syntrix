# Phase 2 — Q&A Layer (Overview)

Per PRD: "Question / answer post type, accepted answer, tags + tag pages,
reputation + badges, markdown polish (code highlighting, math, diagrams),
duplicate detection, answer-shaped feeds."

## Design Principles

Q&A is **not** a separate app — it's a new post type within existing
communities. A community can have both discussion posts and question posts.
The same shell, sidebar, feeds, voting, and commenting infrastructure is
reused. New pieces:

- **Questions** are posts with `post_type = 'question'` and a tag set
- **Answers** are a new first-class entity (not comments) — votable,
  one can be "accepted," sorted by score
- **Tags** are community-scoped labels for categorizing questions
- **Reputation** is a read-model derived from activity (votes received,
  answers accepted) — not a currency
- **Badges** are milestone achievements awarded by triggers

## Data Model Additions

### New tables (all in `syntrix` schema)

```
tags
  id          uuid PK
  community_id uuid FK → communities  (tags are community-scoped)
  name        citext NOT NULL
  slug        citext NOT NULL
  description text
  color       text
  usage_count int DEFAULT 0  (trigger-maintained)
  created_by  uuid FK → users
  created_at  timestamptz
  UNIQUE(community_id, slug)

question_tags  (M:N join)
  question_id uuid FK → posts
  tag_id      uuid FK → tags
  PK(question_id, tag_id)

answers
  id          uuid PK
  question_id uuid FK → posts (WHERE post_type='question')
  author_id   uuid FK → users (SET NULL on delete)
  body_json   jsonb NOT NULL
  body_html   text NOT NULL
  score       int DEFAULT 0  (trigger-maintained)
  is_accepted bool DEFAULT false
  accepted_at timestamptz
  deleted_at  timestamptz
  removed_at  timestamptz
  removed_by  uuid FK → users
  removed_reason text
  created_at  timestamptz
  updated_at  timestamptz

answer_votes
  id          uuid PK
  answer_id   uuid FK → answers
  user_id     uuid FK → users
  value       smallint CHECK (value IN (-1, 1))
  created_at  timestamptz
  UNIQUE(answer_id, user_id)

reputation_events
  id          uuid PK
  user_id     uuid FK → users
  event_type  text NOT NULL  ('answer_upvoted', 'answer_accepted', etc.)
  delta       int NOT NULL   (+10, +15, -2, etc.)
  source_id   uuid           (the answer/question/post that triggered it)
  created_at  timestamptz

badges
  id          uuid PK
  slug        citext UNIQUE
  name        text NOT NULL
  description text
  icon        text           (emoji or icon name)
  tier        text           ('bronze', 'silver', 'gold')
  criteria    jsonb          (machine-readable award condition)

user_badges
  id          uuid PK
  user_id     uuid FK → users
  badge_id    uuid FK → badges
  awarded_at  timestamptz
  UNIQUE(user_id, badge_id)  -- each badge awarded once
```

### Modifications to existing tables

```
posts
  + post_type  text DEFAULT 'discussion' CHECK (IN ('discussion', 'question'))
  + answer_count int DEFAULT 0  (trigger-maintained, questions only)
  + has_accepted_answer bool DEFAULT false

users
  + reputation  int DEFAULT 0  (trigger-maintained from reputation_events)
```

## Reputation Point Values

| Event | Delta | Reverse |
|-------|-------|---------|
| Your answer upvoted | +10 | -10 on unvote |
| Your answer downvoted | -2 | +2 on unvote |
| Your answer accepted | +15 | -15 on unaccept |
| You accept an answer (asker) | +2 | -2 on unaccept |
| Your question upvoted | +5 | -5 on unvote |
| Your question downvoted | -2 | +2 on unvote |
| You downvote an answer (cost) | -1 | +1 on unvote |

## Key Decisions

1. **Tags are community-scoped**, not global. A "python" tag in the Go
   community is independent of "python" in the SRE community. This keeps
   tag curation manageable per community.

2. **Answers are not comments.** They're a separate table with their own
   voting, accepted state, and sorting. Comments on answers use the
   existing `comments` table (with a new nullable `answer_id` FK or by
   pointing `post_id` to the question and using a discriminator).

3. **Reputation is eventually consistent.** `reputation_events` is the
   event log; `users.reputation` is a denormalized total maintained by
   triggers. If it drifts, a reconciliation query can recompute.

4. **Badges are checked lazily** — on relevant actions (post answer,
   receive vote), not via a batch job. A trigger or post-action hook
   checks badge criteria and inserts into `user_badges` if met.

5. **Duplicate detection uses `pg_trgm`** — the extension is already
   installed. `similarity(title, $1)` with a GIN index on `title` gives
   us "similar questions" without Elasticsearch.
