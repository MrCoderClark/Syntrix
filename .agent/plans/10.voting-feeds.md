# Section 10 — Voting & Feeds Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Complexity:** ⚠️ **Medium** — two vote tables with triggers, three feed algorithms (Hot/New/Top), home feed aggregation, and wiring vote widgets into existing post + comment UIs. Well-defined but touches many existing files.

**Goal:** Build the upvote/downvote system for posts and comments, with Postgres triggers maintaining `score` counters atomically. Implement Hot/New/Top feed algorithms. Wire vote widgets into the existing post card, post detail, and comment UIs. Build the "Your Feed" home view that aggregates posts from the user's joined communities.

**Architecture:**

- **Two vote tables:** `post_votes(user_id, post_id, value)` and `comment_votes(user_id, comment_id, value)`. `value` is `1` (upvote) or `-1` (downvote). `UNIQUE(user_id, post_id)` / `UNIQUE(user_id, comment_id)`. Voting uses `INSERT ... ON CONFLICT DO UPDATE SET value = EXCLUDED.value`. Un-voting is `DELETE`.
- **Triggers:** AFTER INSERT/UPDATE/DELETE on each vote table → recompute `posts.score` / `comments.score` as `SUM(value)`. Atomic with the vote write — no app math.
- **Cannot vote on own content.** Enforced in the API handler (not a DB constraint, since it requires joining to the content table).
- **Hot algorithm:** Adaptation of Reddit's hot ranking: `sign(score) * log10(max(|score|, 1)) + (created_at_epoch - reference_epoch) / 45000`. Higher scores and newer posts rank higher. Computed as a functional index on `posts` for efficient sorting.
- **New:** `ORDER BY created_at DESC`. Already works with existing index `(community_id, created_at DESC)`.
- **Top:** `ORDER BY score DESC, created_at DESC`. Period filters: Today, This Week, This Month, All Time.
- **Home feed ("Your Feed"):** Posts from all communities the user has joined, sorted by Hot/New/Top. Uses a JOIN through `community_memberships`. For anonymous users, shows global Hot feed.
- **Rate limit:** 600 votes/hour per user (from PRD).

**Tech Stack:** SQLAlchemy Core, Postgres triggers, Pydantic, React (optimistic UI).

- [ ] **Step 2: Generate Alembic migration**

The migration must also create:

**Post score trigger:**
```sql
CREATE OR REPLACE FUNCTION syntrix.update_post_score()
RETURNS TRIGGER AS $$
BEGIN
    IF TG_OP = 'INSERT' THEN
        UPDATE syntrix.posts SET score = score + NEW.value WHERE id = NEW.post_id;
        RETURN NEW;
    ELSIF TG_OP = 'DELETE' THEN
        UPDATE syntrix.posts SET score = score - OLD.value WHERE id = OLD.post_id;
        RETURN OLD;
    ELSIF TG_OP = 'UPDATE' THEN
        UPDATE syntrix.posts SET score = score - OLD.value + NEW.value WHERE id = NEW.post_id;
        RETURN NEW;
    END IF;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_post_score
AFTER INSERT OR UPDATE OR DELETE ON syntrix.post_votes
FOR EACH ROW EXECUTE FUNCTION syntrix.update_post_score();
```

**Comment score trigger** (identical pattern for `comment_votes` → `comments.score`):
```sql
CREATE OR REPLACE FUNCTION syntrix.update_comment_score()
RETURNS TRIGGER AS $$
BEGIN
    IF TG_OP = 'INSERT' THEN
        UPDATE syntrix.comments SET score = score + NEW.value WHERE id = NEW.comment_id;
        RETURN NEW;
    ELSIF TG_OP = 'DELETE' THEN
        UPDATE syntrix.comments SET score = score - OLD.value WHERE id = OLD.comment_id;
        RETURN OLD;
    ELSIF TG_OP = 'UPDATE' THEN
        UPDATE syntrix.comments SET score = score - OLD.value + NEW.value WHERE id = NEW.comment_id;
        RETURN NEW;
    END IF;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_comment_score
AFTER INSERT OR UPDATE OR DELETE ON syntrix.comment_votes
FOR EACH ROW EXECUTE FUNCTION syntrix.update_comment_score();
```

**Hot score functional index:**
```sql
CREATE INDEX idx_posts_hot ON syntrix.posts (
    (
        sign(score) * log(greatest(abs(score), 1))
        + extract(epoch FROM created_at - '2024-01-01T00:00:00Z'::timestamptz) / 45000.0
    ) DESC
);
```

**Vote flow (post):**
1. Auth + permission check (not own post, not banned).
2. Rate limit: 600/hr per user.
3. If `value == 0`: `DELETE FROM post_votes WHERE user_id = $1 AND post_id = $2`. Trigger decrements score.
4. If `value == 1 or -1`: `INSERT INTO post_votes (user_id, post_id, value) VALUES ($1, $2, $3) ON CONFLICT (user_id, post_id) DO UPDATE SET value = EXCLUDED.value`. Trigger adjusts score.
5. Fetch updated `posts.score`.
6. Return `VoteResponse(score=..., user_vote=value)`.

Comment voting is identical pattern.

## Task 5 — Feed algorithms

- [ ] **Step 1: Create `backend/app/feeds/algorithms.py`**

```python
from __future__ import annotations

from datetime import datetime, timedelta, UTC
from sqlalchemy import select, desc, func, text, and_

from app.posts.models import posts
from app.comments.models import comments


REFERENCE_EPOCH = datetime(2024, 1, 1, tzinfo=UTC)


def hot_score_expr():
    """SQL expression for hot ranking score."""
    score = posts.c.score
    age = func.extract("epoch", posts.c.created_at - REFERENCE_EPOCH)
    return (
        func.sign(score) * func.log(func.greatest(func.abs(score), 1))
        + age / 45000.0
    )


def new_order():
    return [desc(posts.c.created_at), desc(posts.c.id)]


def hot_order():
    return [desc(hot_score_expr()), desc(posts.c.id)]


def top_order():
    return [desc(posts.c.score), desc(posts.c.created_at), desc(posts.c.id)]


def top_period_filter(period: str):
    """Return a WHERE clause filtering by time period."""
    now = datetime.now(UTC)
    cutoffs = {
        "today": now - timedelta(days=1),
        "week": now - timedelta(weeks=1),
        "month": now - timedelta(days=30),
        "all": None,
    }
    cutoff = cutoffs.get(period)
    if cutoff is None:
        return None
    return posts.c.created_at >= cutoff
```

## Self-review notes

- **Triggers, not app math.** Score recomputation happens atomically inside the vote transaction. There is no window where `posts.score` is out of sync with `SUM(post_votes.value)`. This is the single most important correctness property in the voting system.
- **`ON CONFLICT DO UPDATE`** handles vote changes (upvote → downvote) in one statement. The UPDATE trigger correctly adjusts the score by `NEW.value - OLD.value`. No delete-then-insert race.
- **Un-vote is DELETE, not value=0.** The vote table only stores `-1` and `1`. Un-voting deletes the row. The `VoteRequest` accepts `value=0` as syntactic sugar for "remove my vote" — the router translates this to a DELETE.
- **Hot algorithm matches Reddit's classic formula.** The `45000` divisor means a post gains one "point equivalent" every 12.5 hours. This balances recency against engagement — a brand-new post with 1 vote outranks a day-old post with ~7 votes.
- **Functional index for Hot.** The `idx_posts_hot` index pre-computes the hot score so `ORDER BY hot_score DESC` is an index scan, not a full-table computation. This is critical for feed performance as post count grows.
- **Cannot vote on own content** is enforced at the API layer, not the DB layer. A CHECK constraint would require a JOIN in the constraint, which Postgres doesn't support. The `can_vote_post` / `can_vote_comment` permission functions handle this.
- **Optimistic UI with rollback.** The VoteWidget updates score and arrow state immediately on click, then sends the API call. On error, it reverts to the previous state. This makes voting feel instant.
- **Cursor-based pagination.** No offset pagination (which gets slower as page number grows). The cursor encodes the last-seen sort value + id, making every page fetch O(1) via index seek.
