# Section 08 — Posts Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Complexity:** 🔴 **Complex** — TipTap editor integration, markdown source-toggle, server-side rendering of TipTap JSON → HTML, OG link previews, post CRUD with image attachment lifecycle, mod remove. This is the heaviest section in Phase 1.

**Goal:** Build the full post creation, editing, viewing, and moderation pipeline: a TipTap-based rich editor with code blocks (Shiki), inline images (from Section 07 upload flow), link previews, markdown source-toggle; backend post CRUD with body_json → body_html server rendering; post detail page; community feed; mod remove with reason.

**Architecture:**

- **Editor:** TipTap with extensions: StarterKit (bold/italic/lists/headings/blockquote/code), Image (inline, from Section 07 signed uploads), CodeBlockShiki (syntax highlighting), Link (with OG preview card), Table. A "Source" toggle swaps to a `<textarea>` showing markdown (converted via `tiptap-markdown`). Canonical storage is TipTap JSON in `posts.body_json` (jsonb). Server pre-renders to `posts.body_html` on every create/edit.
- **Server rendering:** A thin Python function that walks the TipTap JSON tree and emits HTML. This cached `body_html` is what read paths serve — no client-side re-rendering of JSON for readers.
- **Image lifecycle (Band B):** On post submit, FastAPI extracts image URLs from `body_json`, matches them against `pending/` keys, INSERTs `post_attachments`, moves objects from `pending/` → `posts/<post_id>/<sha>.ext`. DB writes first; storage mutations follow.
- **Mod remove:** Sets `removed_at` + `removed_by` + `removed_reason`. Distinct from author soft-delete (`deleted_at`). UI shows "[removed by mod]" vs "[deleted by author]".
- **OG link previews:** On paste of a URL, the editor shows an inline preview card. FastAPI has a `/api/og` endpoint that fetches and parses OG meta tags. Results are cached in a `og_cache` dict (in-memory, no Redis in Phase 1).

**Tech Stack:** TipTap, @tiptap/extension-image, @tiptap/extension-link, @tiptap/extension-code-block-lowlight (with Shiki adapter), tiptap-markdown, Pydantic, SQLAlchemy Core.

**Create post flow (Band B from PRD):**
1. Validate auth + membership + rate limit.
2. Parse `body_json`, render to `body_html` via renderer.
3. Extract image URLs from `body_json` nodes.
4. In a transaction: INSERT post, INSERT `post_attachments` for each image.
5. After commit: move each image from `pending/` → `posts/<post_id>/<sha>.ext` via `StorageBackend.move()`.
6. Return `PostResponse`.

**Pagination:** cursor-based using `(created_at, id)` for New feed. Sort param selects Hot/New/Top (Hot algorithm in Section 10).

## Task 5 — OG link preview endpoint

- [ ] **Step 1: Create `backend/app/posts/og.py`**

```python
from __future__ import annotations

import asyncio
from dataclasses import dataclass

import httpx
from fastapi import APIRouter

router = APIRouter(prefix="/api/og", tags=["og"])

_og_cache: dict[str, OGData | None] = {}


@dataclass
class OGData:
    title: str | None
    description: str | None
    image: str | None
    site_name: str | None
    url: str


async def fetch_og(url: str) -> OGData | None:
    if url in _og_cache:
        return _og_cache[url]

    try:
        async with httpx.AsyncClient(timeout=5.0, follow_redirects=True) as client:
            resp = await client.get(url, headers={"User-Agent": "Syntrix/1.0 bot"})
            resp.raise_for_status()
            # Parse OG meta tags from HTML
            # Simple regex or html.parser — no heavy dep
            html_text = resp.text[:50_000]  # cap parsing
            data = _parse_og_tags(html_text, url)
            _og_cache[url] = data
            return data
    except Exception:
        _og_cache[url] = None
        return None


def _parse_og_tags(html: str, url: str) -> OGData:
    import re
    def get_og(prop: str) -> str | None:
        match = re.search(
            rf'<meta\s+(?:property|name)=["\']og:{prop}["\']\s+content=["\']([^"\']*)["\']',
            html, re.IGNORECASE
        )
        return match.group(1) if match else None

    return OGData(
        title=get_og("title"),
        description=get_og("description"),
        image=get_og("image"),
        site_name=get_og("site_name"),
        url=url,
    )


@router.get("")
async def get_og_data(url: str):
    data = await fetch_og(url)
    if data is None:
        return {"url": url, "title": None, "description": None, "image": None, "site_name": None}
    return {
        "url": data.url,
        "title": data.title,
        "description": data.description,
        "image": data.image,
        "site_name": data.site_name,
    }
```

## Self-review notes

- **`body_html` is server-rendered once and cached.** Read paths serve pre-rendered HTML. The renderer runs on create and edit only. This avoids shipping TipTap to readers and keeps read performance flat regardless of post complexity.
- **XSS prevention:** The server renderer HTML-escapes all text nodes. Image `src` is validated as a known Supabase Storage URL. Link `href` is rendered as-is but displayed in a `target="_blank" rel="noopener"` tag. The `PostBody` component uses `dangerouslySetInnerHTML` — safe because the HTML is generated server-side from trusted TipTap JSON, not user-supplied raw HTML.
- **Image lifecycle completes here.** Section 07 handled upload → finalize (tmp/ → pending/). This section handles post submit (pending/ → posts/) and post delete (cascade deletes attachments → async storage sweep deletes bytes).
- **Mod remove vs. author delete:** Two separate columns, two separate UI treatments. A mod cannot "edit" a post (that would misrepresent authorship). They can only remove it with a reason.
- **OG cache is in-memory.** No Redis in Phase 1. The cache is per-process and lost on restart. This is acceptable for local development. A TTL eviction or LRU could be added later.
- **Pin constraint:** At most one pinned post per community. Pinning a new post auto-unpins the previous one. This is enforced in the router, not by a DB constraint (to allow the atomic swap).
