# Section 09 — Comments Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Complexity:** ⚠️ **Medium** — ltree threading is well-understood but the tree rendering, reply UI, and depth limits require care. TipTap reuse from Section 08 keeps the editor work light.

**Goal:** Build threaded comments on posts using PostgreSQL's `ltree` extension for efficient subtree fetching. Comments use a lightweight TipTap editor (subset of Section 08's extensions). Support reply at any depth (capped at 10), author delete (soft), mod remove with reason, and a flat-then-threaded rendering model (top-level comments sorted by score, replies sorted chronologically).

**Architecture:**

- **Threading model:** Each comment has a `path` column of type `ltree`. Root comments get `path = <comment_id_hex>`. Replies get `path = <parent_path>.<child_id_hex>`. Depth is `nlevel(path)`. Fetching a post's comments is `SELECT * FROM comments WHERE post_id = $1 ORDER BY path` — ltree's lexicographic ordering naturally produces depth-first traversal.
- **Depth cap:** 10 levels. Reply attempts beyond depth 10 return 400. The UI indents up to depth 5, then flattens remaining levels visually (shows "replying to @handle" instead of further indentation).
- **Editor:** A stripped-down TipTap instance — paragraph, bold, italic, code, inline code, link, blockquote. No images, no tables, no headings. Markdown toggle reused from Section 08.
- **Sorting:** Top-level comments (depth 1) are sorted by score descending (after Section 10). Replies within a thread are sorted by `path` (chronological by insertion order). The API returns comments in `path` order; the frontend groups and sorts top-level comments client-side.
- **Delete semantics:** Author soft-delete sets `deleted_at`, replaces body with "[deleted]" in API response but preserves the tree structure (children remain visible). Mod remove sets `removed_at + removed_by + removed_reason`, shows "[removed by moderator]". If all children of a deleted/removed comment are also deleted, the entire subtree is pruned from the response.
- **Counter trigger:** INSERT/DELETE on `comments` maintains `posts.comment_count` via a Postgres trigger (created in this section's migration).

**Tech Stack:** ltree, SQLAlchemy Core, TipTap (subset), Pydantic.

- [ ] **Step 2: Generate Alembic migration**

The migration must also create:
1. Index on `(post_id, path)` using GiST for ltree subtree queries
2. A Postgres trigger function `syntrix.update_comment_count()` that increments `posts.comment_count` on INSERT and decrements on DELETE (only counting non-deleted, non-removed comments)
3. A trigger on `syntrix.comments` AFTER INSERT OR DELETE that calls the function

```sql
CREATE OR REPLACE FUNCTION syntrix.update_comment_count()
RETURNS TRIGGER AS $$
BEGIN
    IF TG_OP = 'INSERT' THEN
        UPDATE syntrix.posts SET comment_count = comment_count + 1 WHERE id = NEW.post_id;
        RETURN NEW;
    ELSIF TG_OP = 'DELETE' THEN
        UPDATE syntrix.posts SET comment_count = comment_count - 1 WHERE id = OLD.post_id;
        RETURN OLD;
    END IF;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_comment_count
AFTER INSERT OR DELETE ON syntrix.comments
FOR EACH ROW EXECUTE FUNCTION syntrix.update_comment_count();
```

- [ ] **Step 2: Implement `backend/app/comments/tree.py`**

```python
from __future__ import annotations

MAX_DEPTH = 10


def compute_path(parent_path: str | None, comment_id_hex: str) -> str:
    if parent_path is None:
        return comment_id_hex
    depth = parent_path.count(".") + 2  # parent depth + 1
    if depth > MAX_DEPTH:
        raise ValueError(f"Comment depth {depth} exceeds max {MAX_DEPTH}")
    return f"{parent_path}.{comment_id_hex}"


def depth_from_path(path: str) -> int:
    return path.count(".") + 1


def build_comment_tree(rows: list[dict]) -> list[dict]:
    """Build a nested tree from ltree-sorted flat rows."""
    nodes: dict[str, dict] = {}
    roots: list[dict] = []

    for row in rows:
        node = {**row, "children": []}
        nodes[row["path"]] = node

        if row["depth"] == 1:
            roots.append(node)
        else:
            parent_path = row["path"].rsplit(".", 1)[0]
            parent = nodes.get(parent_path)
            if parent:
                parent["children"].append(node)
            else:
                roots.append(node)

    return roots
```

## Self-review notes

- **ltree is the right tool.** `path` ordering gives us depth-first traversal for free — no recursive CTEs, no adjacency-list joins. A single `WHERE post_id = $1 ORDER BY path` query returns the entire tree in display order.
- **Depth cap of 10 prevents abuse.** Deeply nested threads are unreadable. The visual indent caps at 5 to keep comments readable on narrow screens; deeper replies show "replying to @handle" context.
- **Comment counter trigger is atomic.** The trigger runs inside the INSERT transaction, so `posts.comment_count` can never desync from reality. No app-level counter math.
- **Deleted comments preserve tree structure.** A deleted parent with live children remains as a "[deleted]" placeholder. Only if the entire subtree is dead do we omit it from the response. This matches Reddit/HN behavior.
- **The comment editor is intentionally limited.** No images, tables, or headings in comments. This keeps comment threads scannable and prevents walls of formatted content.
- **`CommentResponse` is self-referential** (`children: list[CommentResponse]`). Pydantic V2 handles this with `model_rebuild()`.
