Design Section 2 / 5 — Data Model

The shape of Syntrix

Twelve tables, grouped into four bands: identity, structure, content, engagement. Postgres only — no Redis, no Mongo, no Elasticsearch. The non-obvious choices (TipTap in jsonb, soft-vs-hard delete, denormalized counters) are called out below the diagram.

The non-obvious choices

01TipTap JSON in jsonb — server pre-renders body_html

Posts and comments both store the canonical TipTap document as jsonb. On every create/edit, FastAPI renders that JSON to HTML and saves it to a body_html column. Read paths never re-render — they serve the cached HTML.

Why Round-trips perfectly (canonical) AND fast SSR (cached). Markdown source-toggle just deserializes the JSON, not the HTML.

02Soft delete vs. hard delete

Posts and comments use soft delete (deleted_at timestamp) so moderator audit history holds and accidental deletes have a 24h reversal window. Attachments use hard delete via ON DELETE CASCADE + an async MinIO sweep.

Why Soft delete preserves graph integrity (orphaned comment threads are awkward); hard delete on binaries keeps storage bounded.

03Mod removal  ≠  user delete

Two separate columns: deleted_at (user removed it) and removed_at + removed_by (mod removed it). The UI treats them differently — "[deleted by author]" vs "[removed by mod for: spam]".

Why Conflating them loses information mods, users, and audit logs all need.

04Denormalized counters via Postgres triggers

posts.score, posts.comment_count, comments.score are kept up-to-date by Postgres triggers on the vote and comment tables. App code reads the cached value; it never sums on the fly.

Why Triggers are atomic with the change. App-level "update counter then write" is two queries and a race.

05Two vote tables, not one polymorphic

post_votes and comment_votes are separate so we keep real foreign keys to posts.id and comments.id. A single votes(target_type, target_id) table loses FK enforcement.

Why Database integrity > one fewer table.

06Refresh tokens are revocable (no sessions table)

Access JWT lives 15 min, refresh token (opaque, hashed) lives 30 days in refresh_tokens. "Log out everywhere" = mark all rows revoked_at = now(). No separate sessions table — refresh tokens *are* the session.

Why Common pattern, no extra table, supports the operations we need.

07citext + pg_trgm extensions

citext for users.handle and communities.slug — case-insensitive uniqueness without losing original case. pg_trgm for fuzzy search later (Phase 2). ltree on comments.path for fast subtree fetches.

Why Postgres has the right tool. Use it instead of normalizing to lowercase or adding Elasticsearch.

08What's not in Phase 1

  • notifications — UI stub only; no table yet. Deferred to Phase 2.
  • tags — tags on posts come with Q&A (Phase 2).
  • mod_action UI — table exists, logged backend-side, no admin view.
  • follows / friends — no follow graph in Phase 1.
  • messages / DMs — Phase 3.

Index strategy (what's indexed and why)

  • posts (community_id, created_at desc) — per-community "New" feed
  • posts (community_id, score desc, created_at desc) — per-community "Top" feed
  • posts ((computed_hot_score) desc) — global "Hot" feed (functional index on a Reddit-style formula)
  • comments (post_id, path) — fetch comment tree fast
  • post_votes (post_id), comment_votes (comment_id) — for counter recompute jobs
  • oauth_identities (provider, provider_sub) unique — OAuth login lookup
  • communities (slug) unique, users (handle) unique — citext
  • refresh_tokens (user_id) where revoked_at is null — partial index for active sessions
