Who can do what

Five roles. One matrix. Plus a rate-limit policy on the spam-prone surfaces — implemented in Postgres (no Redis in Phase 1). Everything below is enforced at the FastAPI layer; the edge never makes a permission decision.

Implementation notes

Authz lives in app/auth/perm.py

  • One function per action: can_create_post(user, community), can_remove_post(user, post), etc. Easy to grep, easy to test.
  • FastAPI dependency: Depends(can("create_post", community_id)) raises 403 on deny.
  • No "if user.is_admin" sprinkled through handlers. The permission function is the only place that knows.
  • Membership + role + ban queried once, cached per-request — one round-trip max.

Rate limits live in Postgres (no Redis)

  • One table: rate_limit_buckets(key, tokens, refilled_at). Key = "post_create:user:<uuid>", "login:ip:<cidr>", etc.
  • Atomic refill + take in a single UPDATE … RETURNING with a WHERE tokens >= cost guard. Returns 0 rows when the bucket is empty → 429.
  • Janitor job sweeps stale rows nightly.
  • Redis can come later if we ever outgrow this. Phase 1: no extra service.

403 vs 404 vs 429

  • 403 "forbidden" when the caller exists but isn't allowed (denied by perm.py).
  • 404 "not found" when the resource exists but the caller isn't allowed to know it exists (rare in Phase 1 — public communities don't hide).
  • 429 "too many requests" with a Retry-After header on rate limit. Body includes the next-bucket-refill timestamp.

Vote self-ban & ban semantics

  • You cannot vote on your own post or comment. Returns 403 with code self_vote.
  • If a user is banned from a community, they keep read access but lose post / comment / vote rights. UI surfaces the ban explicitly ("you were banned from c/halo — reason: …").
  • Site-wide suspension (users.is_suspended = true) kills all write rights everywhere; reads still work (the user can see why they were suspended).

What's deliberately NOT in this section

  • Report / flag flow — deferred to Phase 2. For now, mods discover bad content by reading it. Reports turn the discovery into a feed.
  • Audit log UI — mod_actions rows are written, but no admin view in Phase 1. Query the table if you need to audit.
  • Automod rules / regex bans / link blacklists — Phase 2 at the earliest.
  • Captcha — not in Phase 1. Rate limits + OAuth gating (no email-password) eliminate most of the spam vectors that captchas defend against.
