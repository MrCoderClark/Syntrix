Authz model — defense in depth

Corrected from v1

Two layers, one source of truth

v1 said "all authz lives in FastAPI." More precisely: all permission logic lives in FastAPI, but authentication is checked at both layers. This is the production-grade pattern — the edge catches garbage requests fast; the backend is the authority on what each caller is allowed to do.

Edge — Next.js middleware

Authentication only

  • Verify JWT signature (shared secret or public key)
  • Check expiration
  • Redirect unauth'd users from protected routes
  • No permission logic. No DB calls.
Backend — FastAPI

Authentication + Authorization

  • Re-verify JWT (never trust the edge)
  • Resource-level checks: ownership, mod role, community membership, ban status
  • Every protected route uses a Depends(current_user) + Depends(can(action, resource)) pattern
  • Source of truth for every "are they allowed to" question

Image lifecycle — permanent once linked

Clarified from v1

Direct-to-Supabase Storage doesn't compromise durability

Direct upload changes who carries the bytes, not how long they persist. Once an image is linked to a saved post, it lives as long as the post does. Deletion is explicit only — driven by post edit or post delete.

1Browser asks FastAPI to sign an upload. POST /uploads/sign returns {key, url, expires_in: 300}. Key points at tmp/<random>.jpg. URL is good for 5 minutes only. 2Browser uploads directly to Supabase Storage via PUT to the signed URL. Bytes never touch FastAPI. 3Browser confirms with FastAPI. POST /uploads/finalize {key}. FastAPI: opens the object, validates content-type + dimensions (rejects on mismatch), strips EXIF, moves it from tmp/ to posts/<post-id>/<sha>.jpg, returns the canonical URL. 4Editor embeds the canonical URL in the TipTap doc. 5On post submit, FastAPI parses the TipTap JSON, extracts every image URL, and writes post_attachments rows linking (post_id, object_key). The image is now permanent.

Deletion happens only when:

  • The post is deleted → ON DELETE CASCADE on post_attachments → background task deletes the Supabase Storage objects.
  • The post is edited and an image is removed from the TipTap doc → on save, FastAPI diffs the new attachment set against the old one and deletes orphans.
  • A tmp/ object lives past 24 h without being finalized → swept by a daily TTL cleanup job (it never reached step 5, so it's nobody's image).

Note on outages: if Supabase Storage is down, the image is unreachable — but that's true whether FastAPI proxies the bytes or not. Production durability (backups, versioning, CDN cache) is a hosting decision deferred until we leave local-only.

What's deliberately NOT here

No Redis (FastAPI in-process cache is enough for Phase 1). No background worker / Celery (every Phase 1 action fits a single request; the 24h orphan-sweep is a cron / scheduled task, not a worker). No CDN (local dev — production decides later). No analytics / metrics service (structured logs only). Each of those gets added in the phase that actually needs it, not preemptively.
