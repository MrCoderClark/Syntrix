Design Section 3 / 5 — Key Flows

How Syntrix moves

Four flows that exercise every interesting edge: OAuth signin, post-with-image upload (the most complex one), vote (with trigger-driven counter), and comment. Each shows who talks to whom, in what order, and which steps are authorized where.

Flow 01

OAuth signin — identity in 7 hops

What's load-bearing about this flow

  1. state is generated by FastAPI, signed (HMAC), and stashed in a short-lived cookie. Round-trips through GitHub. Validated on callback — catches CSRF.
  2. Provider's access token is never persisted — used once on the callback to fetch identity, then discarded.
  3. UPSERT key is (provider, provider_sub) not email — emails change, sub doesn't.
  4. Refresh cookie + access cookie are both httpOnly + Secure + SameSite=Lax. Access is 15 min, refresh is 30 days, refresh is opaque + hashed in DB.
  5. If the email matches an existing Syntrix user with a *different* provider, we surface "link existing account?" — never silent-merge.
Endpoints: GET /api/auth/login/<provider> · GET /api/auth/callback/<provider> · POST /api/auth/refresh · POST /api/auth/logout
Flow 02 — the hard one

Create a post with an image

Why this flow has two bands

  1. The image upload happens before the post exists. Bytes go straight from browser to Supabase Storage. FastAPI only signs and finalizes — wrapped behind a StorageBackend interface so the app is portable.
  2. After finalize, the image lives in pending/ — it's been validated but isn't linked to anything. If the user closes the browser tab here, the orphan sweep deletes it after 24h.
  3. At submit time, FastAPI re-parses the TipTap JSON, extracts every image URL, inserts post_attachments rows, then moves the objects from pending/ to posts/<post-id>/<sha>.jpg. The path now contains the post-id so the lifecycle binding is unambiguous.
  4. All storage mutations happen after the DB writes succeed — if the DB transaction rolls back, the bytes stay in pending/ and get swept normally. No orphaned attachment rows pointing at missing objects.
  5. title server-validates ≤200 chars; body_json server-validates against a TipTap schema allowlist (rejects unknown node types).
Endpoints: POST /api/uploads/sign · POST /api/uploads/finalize · POST /api/posts
Flow 03

Vote — triggered counter, no app math

  1. Browser → FastAPI: POST /posts/<id>/vote {value: 1}.
  2. FastAPI: authz — logged in, post not removed, user not banned from community.
  3. FastAPI → Postgres: INSERT INTO post_votes … ON CONFLICT (user_id, post_id) DO UPDATE SET value = EXCLUDED.value.
  4. Postgres trigger after_post_vote_change: recomputes posts.score for that row.
  5. FastAPI returns {score, user_vote}; UI confirms the optimistic update.
Why triggers: +1 and counter increment land in one atomic transaction. Why ON CONFLICT: users flip votes; an upsert handles all three states (up, neutral, down).
Flow 04

Comment — TipTap, threading, counter

  1. Browser → FastAPI: POST /posts/<id>/comments {body_json, parent_comment_id?}.
  2. FastAPI: authz (member, not banned). Render TipTap JSON → HTML server-side. Compute depth + path (ltree) from parent.
  3. FastAPI → Postgres: INSERT INTO comments.
  4. Trigger after_comment_insert: posts.comment_count++ on the parent post.
  5. FastAPI returns the comment with rendered HTML; UI inserts it at the right depth in the live tree.
Image in a comment? Same upload flow as posts (Flow 02), just with comment_attachments. Identical lifecycle.

Things every flow shares

  1. Auth at the edge first. Every browser-originated request passes Next.js middleware (JWT signature + expiry) before reaching FastAPI. Unauth'd requests get 401 before they cost backend time.
  2. FastAPI re-verifies the JWT. Never trust the edge.
  3. Resource authz is its own dependency. Pattern: Depends(current_user) then Depends(can("create_post", community_id)). The "can" helper hits cached community membership/role.
  4. Writes use serializable transactions on Postgres. Reads use the default isolation. Triggers run inside the write transaction so the counter update can't desync.
  5. Error shape is uniform. All API errors return {error: {code, message, details?}} with a stable code the frontend dispatches on.
