of Reddit (communities + posts), Stack Overflow (Q&A), and Discord (real-time
chat). Strong, distinctive visual identity is a first-class requirement, not a
polish step.

## Stack

- **Backend:** Python + FastAPI (the "brain")
- **Frontend:** Next.js (App Router) + React + TypeScript. Hand-rolled CSS with CSS variables (no Tailwind utility-flood). Fraunces + Hanken Grotesk + JetBrains Mono via `next/font`.
- **Database:** PostgreSQL — **reused from the user's local Supabase Docker stack**. All Syntrix tables live in the `syntrix` schema. Other Supabase schemas (`public`, `auth`, `storage`, `realtime`, `extensions`) are **off-limits** to our migrations and queries.

## Database isolation — STRICT

This database is shared with the user's other Supabase-resident data. Treat it
as a hosted multi-tenant store; we are one tenant in the `syntrix` schema.

- **Schema:** `syntrix`. Every table, view, function, and index goes here.
- **Role:** dedicated `syntrix_app` Postgres user. The app config never carries
  Supabase superuser keys. Grants are `USAGE, CREATE ON SCHEMA syntrix` plus
  table-level privileges as needed. Migrations may need a separate `syntrix_admin`
  role with `CREATE EXTENSION` privileges if extensions aren't pre-installed.
- **Search path:** `search_path = syntrix, public` set per-connection in SQLAlchemy.
- **Alembic:** configured with `version_table_schema = "syntrix"` (the migrations
  table lives inside our schema). An `include_object` filter in `env.py` must
  refuse to operate on any non-`syntrix` schema — safety belt against an
  autogenerated migration ever touching `auth.users` or similar.
- **Connection DSN:** `postgresql+psycopg://syntrix_app:***@127.0.0.1:5432/postgres`
  (via Supavisor pooler).
- **Extensions:** `citext`, `pg_trgm`, `ltree`, `pgcrypto`. `CREATE EXTENSION IF
  NOT EXISTS` — most are already in Supabase's stack.
- **Storage backend (Phase 1):** Supabase Storage (reused container) wrapped
  behind a `StorageBackend` interface so the app stays portable to other
  S3-compatible providers. Bucket: `syntrix-uploads`.
- **Realtime (Phase 3):** own WebSocket gateway — standalone Starlette app
  (`gateway/`, port 8002) with Redis pub/sub. Not Supabase Realtime.

When Claude generates a migration, always inspect it before applying — confirm
every CREATE/ALTER/DROP targets `syntrix.*`. If autogen produces an op that
touches another schema, that's a bug in the `include_object` filter, not an
acceptable migration.
- **Post editor:** TipTap (canonical storage = TipTap JSON); Shiki for code highlighting; markdown as paste/source-toggle alternate
- **Object storage:** **Supabase Storage** — reused from the local Supabase Docker stack. Bucket `syntrix-uploads`. Wrapped behind a `StorageBackend` interface (methods: `sign_upload(key, mime, ttl)`, `finalize(key) -> (canonical_url, metadata)`, `move(src, dst)`, `delete(key)`). One implementation now (`SupabaseStorageBackend`); future implementations (`S3Backend`, `R2Backend`) can swap in without changing app code.
- **Hosting:** all local — backend on port 8001, gateway on port 8002, frontend on port 3000 (all via `uv run uvicorn` / `npm run dev`). Supabase + Redis stay on Docker. Production deployment is a post-Phase 1 decision — code stays 12-factor so it can move anywhere.
- **Auth:** FastAPI owns identity. OAuth-only (GitHub, Google, Discord) via `authlib`. JWT in httpOnly + Secure + SameSite=Lax cookies. No password storage in Phase 1.
- **Authz (defense in depth):** Next.js middleware verifies JWT signature + expiration only (no permission logic, no DB calls). FastAPI re-verifies the JWT and enforces every resource-level check (ownership, mod role, community membership, ban). Permission logic lives in one place: FastAPI.
- **Image lifecycle:** browser uploads directly to Supabase Storage via short-lived signed URLs (5 min). Bytes never transit FastAPI. After upload, the browser pings `/uploads/finalize` — FastAPI validates content-type, strips EXIF, moves the object from `tmp/` to a permanent path, returns the canonical URL. On post submit, `post_attachments` links the object to the post. Images are deleted only on post-delete (cascade), on edit when removed from the TipTap doc, or as orphan-sweep of `tmp/` after 24h.
- **Realtime (Phase 3):** Standalone Starlette WebSocket gateway (`gateway/`,
  port 8002). Redis pub/sub for cross-service messaging. Gateway verifies JWT
  independently (same secret), manages connections in-memory, tracks presence in
  Redis. Backend publishes events; gateway pushes to clients. Redis runs in the
  Supabase Docker stack (`supabase-redis`, port 6379).

## Rules

- Python backend must use a `venv` virtual environment
- Type hints + Pydantic models on every API boundary
- No ORM heroics — prefer SQLAlchemy Core or raw SQL with a thin query layer
- Every table that holds user-owned data has authorization checks at the query
  layer (no "trust the API handler" shortcuts)
- Frontend must not look AI-generated: no default shadcn theme, no generic SaaS
  hero, no "Lorem ipsum" purple-gradient aesthetic. Visual identity is reviewed
  per section.
- Tests live next to the code they test (`backend/foo/test_foo.py` style)

## Branching & merge workflow — STRICT

The user owns every commit, push, and merge. Claude proposes diffs and opens
branches when told to; the user lands them.

**Granularity:** one branch per *section* within a phase. Sections are defined
in each phase's plan (e.g. `feat/auth`, `feat/profiles`, `feat/communities`,
`feat/posts`, `feat/voting`, `feat/design-system`).

**Hard rule — sequential branches, never overlapping:**

1. Claude works on the current section's branch only.
2. When the section is implementation-complete, Claude stops and hands back.
3. The **user** reviews, commits (if Claude hasn't already), pushes, opens a
   PR, merges into `main`, and deletes the merged branch.
4. **Only after the user confirms the merge landed on `main`** does Claude
   create the next section's branch from updated `main`.

Before creating any new feature branch, run `git fetch` + `git status` to
verify the prior section's merge is on `main`. If it isn't, surface that and
wait.

Claude must NOT:
- Create a new feature branch while a previous one is still unmerged.
- Push to `main` directly.
- Merge PRs.
- Force-push to shared branches.
- Skip git hooks (`--no-verify`) or bypass signing.

## Development Flow

1. **Plan** — create a detailed plan and save it to `.agent/plans/`
2. **Branch** — create the section's feature branch from `main`
3. **Build** — execute the plan, following project conventions
4. **Validate** — run tests; for UI work, exercise the feature in a real browser
5. **Hand off** — stop. User commits / pushes / opens PR / merges / deletes
6. **Loop** — only after merge confirmed, branch the next section

## Community model (Phase 1)

- **Admin-curated**: only platform admins create communities. Users can request
  new ones via a form (just creates a `community_request` row; admin approves).
- **Mod powers**: remove post/comment, pin one post, ban user from community.
  No automod, no rules engine, no audit-log UI (logged backend-side only).
- **Owner can rename**, **cannot delete** (admin-only).
- **Private communities deferred to Phase 3** (they couple naturally to private
  chat rooms).
